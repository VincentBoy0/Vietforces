Migration header: "005_daily_completions — tracks which users completed each day's challenge"

Table definition: CREATE TABLE IF NOT EXISTS public.daily_completions with columns:
  - user_id UUID NOT NULL REFERENCES public.users(id) ON DELETE CASCADE
  - challenge_date DATE NOT NULL
  - elo_earned INTEGER NOT NULL DEFAULT 50
  - completed_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
  - PRIMARY KEY (user_id, challenge_date)

Enable RLS: ALTER TABLE public.daily_completions ENABLE ROW LEVEL SECURITY;

RLS policies:
  - "completions_select_own": SELECT USING (user_id = auth.uid()) — users read own rows for history display
  - "completions_service_insert": INSERT TO service_role WITH CHECK (TRUE) — edge functions insert via service_role; the award_daily_bonus function (SECURITY DEFINER) will bypass RLS anyway

Create supabase/migrations/006_daily_bonus_elo.sql with the award_daily_bonus function:

Migration header: "006_daily_bonus_elo — SECURITY DEFINER function that awards +50 bonus ELO for completing daily challenge"

Function signature: CREATE OR REPLACE FUNCTION public.award_daily_bonus(p_user_id UUID, p_challenge_date DATE) RETURNS INTEGER LANGUAGE plpgsql SECURITY DEFINER SET search_path = public AS $$

Function body logic (in PL/pgSQL, NOT a fenced block — describe the steps):
  1. DECLARE v_already_completed BOOLEAN; v_elo_earned INTEGER := 50;
  2. SELECT EXISTS(SELECT 1 FROM daily_completions WHERE user_id = p_user_id AND challenge_date = p_challenge_date) INTO v_already_completed;
  3. IF v_already_completed THEN RAISE EXCEPTION 'already_completed' USING HINT = 'Daily challenge already completed for this date'; END IF;
  4. UPDATE user_progress SET elo_score = elo_score + v_elo_earned, updated_at = NOW() WHERE user_id = p_user_id;
  5. UPDATE leaderboard SET elo_score = elo_score + v_elo_earned, weekly_elo = weekly_elo + v_elo_earned, updated_at = NOW() WHERE user_id = p_user_id;
  6. INSERT INTO daily_completions (user_id, challenge_date, elo_earned) VALUES (p_user_id, p_challenge_date, v_elo_earned);
  7. RETURN v_elo_earned;
  $$ END function

After function definition add:
  REVOKE ALL ON FUNCTION public.award_daily_bonus(UUID, DATE) FROM PUBLIC;
  GRANT EXECUTE ON FUNCTION public.award_daily_bonus(UUID, DATE) TO authenticated;

Both migration files must follow the idempotent style of 001_initial_schema.sql (CREATE TABLE IF NOT EXISTS, CREATE OR REPLACE FUNCTION).

This function is invoked via pg_cron HTTP call (no auth header from pg_cron, but verify using a shared secret via CRON_SECRET env var: check Authorization header = "Bearer " + Deno.env.get("CRON_SECRET")).

Function logic:
  1. Create Supabase client using SUPABASE_URL + SUPABASE_SERVICE_ROLE_KEY (service_role bypasses RLS for insert).
  2. Compute today's UTC date string: new Date().toISOString().split('T')[0]
  3. Check if a challenge already exists for today (SELECT from daily_challenges WHERE challenge_date = today) — if exists, return 200 with "already exists" message (idempotent).
  4. VOCABULARY_POOL: Define a const array of 154 vocabulary IDs that mirrors VocabularyRepository.allVocabulary. Include at minimum: "animal_001" through "animal_020", "school_001" through "school_010", and populate the remaining IDs as a static array (the executor should read VocabularyRepository.kt to extract the full list of all 154 `id` values).
  5. Shuffle VOCABULARY_POOL using Fisher-Yates and take the first 10 IDs → vocabularyIds array.
  6. Pick a random game_mode from ['image_to_word', 'word_to_image', 'fill_blank', 'sentence_order', 'syllable_match'] (these are the modes the DailyChallengeScreen will support).
  7. Insert into daily_challenges: { challenge_date: today, game_mode, vocabulary_ids: vocabularyIds, bonus_elo: 50 }
  8. Return 200 JSON { success: true, date: today, word_count: 10 }

Create supabase/functions/send-streak-reminder/index.ts:

Same structure (Deno.serve, corsHeaders, CRON_SECRET auth check).

Function logic (per NOTIF-02):
  1. Create Supabase service_role client.
  2. Compute today's UTC date string.
  3. Query for users to notify: SELECT u.id, ft.token FROM public.users u JOIN public.fcm_tokens ft ON ft.user_id = u.id LEFT JOIN public.user_progress up ON up.user_id = u.id WHERE (up.last_practice_date IS NULL OR up.last_practice_date < today) AND ft.token IS NOT NULL AND u.is_banned = FALSE. Use supabase-js rpc or raw SQL via postgrest.
  4. For each user's FCM token, send a FCM HTTP v1 notification using fetch():
     - URL: https://fcm.googleapis.com/v1/projects/{FIREBASE_PROJECT_ID}/messages:send
     - Method: POST, Content-Type: application/json
     - Authorization: Bearer {access_token} (obtained by calling getFirebaseAccessToken())
     - Body: JSON with message.token = fcmToken, message.notification = { title: "🔥 Streak in Danger!", body: "Bạn chưa luyện tập hôm nay! Đừng để mất streak nhé." }, message.data = { screen: "daily_challenge" }
  5. Implement getFirebaseAccessToken() as an async helper:
     - Parse FIREBASE_SERVICE_ACCOUNT_JSON from Deno.env.get("FIREBASE_SERVICE_ACCOUNT_JSON")
     - Build a JWT with claims: iss = client_email, sub = client_email, aud = "https://oauth2.googleapis.com/token", scope = "https://www.googleapis.com/auth/firebase.messaging", iat = now, exp = now + 3600
     - Sign the JWT using RS256 with the private_key from service account JSON via Web Crypto API: crypto.subtle.importKey("pkcs8", pemToDer(private_key), { name: "RSASSA-PKCS1-v1_5", hash: "SHA-256" }, false, ["sign"])
     - POST the signed JWT to https://oauth2.googleapis.com/token with grant_type=urn:ietf:params:oauth:grant-type:jwt-bearer
     - Return the access_token from the response
  6. Implement pemToDer(pem: string): ArrayBuffer — strip the PEM header/footer lines, base64-decode the body: atob(pem.replace(/-----.*?-----/g, '').replace(/\n/g, '')), then return Uint8Array.from(decoded, c => c.charCodeAt(0)).buffer
  7. Log successes and failures per token; return 200 with { notified: N, failed: M }

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| cron→Edge Function | pg_cron HTTP calls arrive without user auth context |
| Edge Function→Supabase DB | service_role key bypasses RLS — must be kept in Supabase secrets only |
| Edge Function→FCM API | service account private key must never be committed to source |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Spoofing | generate-daily-challenge endpoint | mitigate | Verify Authorization: Bearer CRON_SECRET on every invocation; reject 401 if missing or wrong |
| T-04-02 | Spoofing | send-streak-reminder endpoint | mitigate | Same CRON_SECRET guard; reject 401 if missing |
| T-04-03 | Elevation of Privilege | award_daily_bonus RPC | mitigate | SECURITY DEFINER with explicit REVOKE/GRANT; only authenticated users can call it; double-completion guard (RAISE EXCEPTION) |
| T-04-04 | Tampering | award_daily_bonus p_user_id param | mitigate | Function runs as definer — client cannot forge another user's ID because the Android caller passes auth.uid() via Supabase SDK which validates JWT first |
| T-04-05 | Information Disclosure | FIREBASE_SERVICE_ACCOUNT_JSON | mitigate | Store exclusively as Supabase secret (supabase secrets set); never commit to git |
| T-04-06 | Denial of Service | generate-daily-challenge double-insert | accept | Idempotent check (SELECT before INSERT) prevents duplicate rows; pg_cron runs once per day |
| T-04-SC | Tampering | Deno npm/std imports in edge functions | mitigate | Use only Deno std lib + @supabase/supabase-js from esm.sh; no npm installs in this plan |
</threat_model>
