      Migration: 008_admin_schema, Date: current date,
      Description: Admin schema additions — is_admin flag on users + vocabulary words table with RLS.
      Idempotent: yes.

    IMPORTANT: Migration 001 already added is_banned to public.users. Do NOT add is_banned again.
    Only add is_admin:

      ALTER TABLE public.users ADD COLUMN IF NOT EXISTS
        is_admin BOOLEAN NOT NULL DEFAULT FALSE;

    Then create the words table:

      CREATE TABLE IF NOT EXISTS public.words (
        id          BIGSERIAL   PRIMARY KEY,
        word        TEXT        NOT NULL,
        classifier  TEXT        NOT NULL DEFAULT '',
        category    TEXT        NOT NULL DEFAULT 'general',
        image_url   TEXT,
        distractors JSONB       NOT NULL DEFAULT '[]',
        created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
      );

    Enable RLS:
      ALTER TABLE public.words ENABLE ROW LEVEL SECURITY;

    Three RLS policies:
    1. "words_select_public" — FOR SELECT USING (TRUE) — anonymous and authenticated users can read.
    2. "words_admin_write" — FOR INSERT TO authenticated WITH CHECK that
       EXISTS (SELECT 1 FROM public.users WHERE id = auth.uid() AND is_admin = TRUE).
    3. "words_admin_update" — FOR UPDATE TO authenticated USING and WITH CHECK that
       EXISTS (SELECT 1 FROM public.users WHERE id = auth.uid() AND is_admin = TRUE).
    4. "words_admin_delete" — FOR DELETE TO authenticated USING that
       EXISTS (SELECT 1 FROM public.users WHERE id = auth.uid() AND is_admin = TRUE).

    Note: The admin dashboard uses the service role key (bypasses RLS). These RLS policies
    are defensive belt-and-suspenders in case the anon key is ever used directly.

    Add an index for category filtering:
      CREATE INDEX IF NOT EXISTS idx_words_category ON public.words (category);

    No Realtime subscription needed for words table (admin-only writes).

    Export async function middleware(request: NextRequest):
      1. Declare: let supabaseResponse = NextResponse.next({ request })
      2. Create supabase client using createServerClient with anon key and this cookie adapter:
           getAll() { return request.cookies.getAll() }
           setAll(cookiesToSet) {
             cookiesToSet.forEach(({ name, value }) => request.cookies.set(name, value))
             supabaseResponse = NextResponse.next({ request })
             cookiesToSet.forEach(({ name, value, options }) =>
               supabaseResponse.cookies.set(name, value, options)
             )
           }
      3. Call: const { data: { user } } = await supabase.auth.getUser()
         (MUST use getUser() not getSession() — getUser() validates JWT server-side)
      4. const pathname = request.nextUrl.pathname

      Guard block for /admin routes:
        if (pathname.startsWith('/admin')) {
          if (!user) return NextResponse.redirect(new URL('/login', request.url))
          // Check is_admin via the anon client (user is authenticated here):
          const { data: profile } = await supabase
            .from('users')
            .select('is_admin')
            .eq('id', user.id)
            .single()
          if (!profile?.is_admin) {
            return NextResponse.redirect(new URL('/login?error=unauthorized', request.url))
          }
        }

      Redirect authed admin away from /login:
        if (pathname === '/login' && user) {
          const { data: profile } = await supabase
            .from('users').select('is_admin').eq('id', user.id).single()
          if (profile?.is_admin) {
            return NextResponse.redirect(new URL('/admin/vocabulary', request.url))
          }
        }

      return supabaseResponse

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → Next.js Server | Login form POST — email/password crosses trust boundary |
| Next.js Server → Supabase (anon) | Session validation using JWT in HTTP-only cookie |
| Next.js Server → Supabase (service role) | Admin mutations — key never leaves server |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01 | Spoofing | middleware.ts auth check | mitigate | Use getUser() (server-side JWT validation) not getSession() (client-side decode only) |
| T-06-02 | Elevation of Privilege | /admin/* routes | mitigate | Middleware checks is_admin=true from users table on every request; non-admin redirected to /login?error=unauthorized |
| T-06-03 | Information Disclosure | SUPABASE_SERVICE_ROLE_KEY | mitigate | Key defined ONLY in SUPABASE_SERVICE_ROLE_KEY (no NEXT_PUBLIC_ prefix); only imported in src/lib/supabase/admin.ts which is server-only |
| T-06-04 | Tampering | loginAction Server Action | mitigate | Server Action runs server-side; credentials never logged; error messages sanitized (encoded before redirect) |
| T-06-05 | Tampering | npm install packages | mitigate | All packages are established ecosystem libraries (next, @supabase/ssr, recharts, tailwindcss) — verify on npmjs.com before install |
| T-06-SC | Tampering | npm/pip/cargo installs | mitigate | slopcheck all new packages; next@15.3.9, @supabase/ssr@0.12.3 verified as canonical packages |
