  <action>
    Create `supabase/migrations/007_activity_feed.sql` following the idempotency conventions in 001–006 (header comment block, `CREATE TABLE IF NOT EXISTS`, `CREATE POLICY IF NOT EXISTS`, `CREATE OR REPLACE FUNCTION`, `CREATE OR REPLACE TRIGGER`).

    **Table: public.activity_events**
    Columns:
    - `id UUID PRIMARY KEY DEFAULT gen_random_uuid()`
    - `user_id UUID NOT NULL REFERENCES public.users(id) ON DELETE CASCADE`
    - `event_type TEXT NOT NULL` — values: `'daily_completion'`, `'elo_milestone'` (future)
    - `metadata JSONB NOT NULL DEFAULT '{}'::JSONB` — event-specific payload (challenge_date, elo_earned, etc.)
    - `created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()`

    Enable RLS on the table.

    **RLS Policies on activity_events:**

    `activity_events_select_following` FOR SELECT:
    `USING (user_id = auth.uid() OR EXISTS (SELECT 1 FROM public.friendships WHERE follower_id = auth.uid() AND following_id = activity_events.user_id))`
    — Authenticated user can see their own events plus events of users they follow (asymmetric follow = follower_id = current user). This is the primary security gate for the feed.

    `activity_events_service_insert` FOR INSERT TO service_role WITH CHECK (TRUE)
    — Only server-side functions (trigger runs as SECURITY DEFINER) and service_role may insert. No client INSERT policy intentionally.

    **RLS Patches on existing tables (SOCIAL-01, SOCIAL-03 infra):**

    Add `users_select_public_username` ON public.users FOR SELECT USING (TRUE)
    — Allows any authenticated user to read all user rows. Multiple SELECT policies are OR'd in Postgres, so existing `users_select_own` (own-row filter) stays intact for clarity; the new permissive policy subsumes it for SELECT. Required for `searchUsers(query)` ILIKE queries from Android.

    Add `progress_select_public` ON public.user_progress FOR SELECT USING (TRUE)
    — Allows any authenticated user to read any user_progress row. Required for PublicProfileViewModel to load target user's `streak_count` and `total_games` (SOCIAL-03). Existing `progress_select_own` and `progress_update_own` policies are unchanged.

    **Trigger function: on_daily_completion_insert()**

    ```
    CREATE OR REPLACE FUNCTION public.on_daily_completion_insert()
    RETURNS TRIGGER
    LANGUAGE plpgsql
    SECURITY DEFINER
    SET search_path = public
    ```

    Body: INSERT INTO public.activity_events (user_id, event_type, metadata) VALUES (NEW.user_id, 'daily_completion', jsonb_build_object('challenge_date', NEW.challenge_date::TEXT, 'elo_earned', NEW.elo_earned))
    RETURN NEW.

    SECURITY DEFINER is required so the function can INSERT into activity_events despite the service_role-only INSERT policy (trigger runs under function definer, not calling user context).

    **Trigger binding:**
    ```
    CREATE OR REPLACE TRIGGER after_daily_completion_insert
      AFTER INSERT ON public.daily_completions
      FOR EACH ROW EXECUTE FUNCTION public.on_daily_completion_insert();
    ```
    (CREATE OR REPLACE TRIGGER is available from Postgres 14; Supabase Cloud uses Postgres 15+.)

    File header must note: Migration 007, date, description = "activity_events table + daily_completion trigger + public RLS for social features (SOCIAL-01, SOCIAL-03, SOCIAL-04)", Idempotent: yes.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → Supabase RLS | Android app cannot INSERT activity_events directly; trigger + service_role only |
| authenticated user → activity_events SELECT | Feed visibility is scoped to own + followed users via EXISTS subquery on friendships |
| authenticated user → users SELECT | Public username search is intentional; no PII beyond username is exposed |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01-01 | Spoofing | activity_events INSERT | mitigate | No client INSERT policy; SECURITY DEFINER trigger is the only writer |
| T-05-01-02 | Tampering | activity_events trigger | mitigate | SECURITY DEFINER + SET search_path = public prevents search_path hijack |
| T-05-01-03 | Information Disclosure | users SELECT policy (public) | accept | Only username is exposed; no PII (email, fcm_token, is_banned not projected in search queries) |
| T-05-01-04 | Information Disclosure | user_progress SELECT policy (public) | accept | Gamification stats (streak_count, total_games, elo_score) — low sensitivity for academic project |
| T-05-01-05 | Elevation of Privilege | friendships subquery in RLS | mitigate | EXISTS subquery references auth.uid() server-side; cannot be spoofed by client |
</threat_model>
