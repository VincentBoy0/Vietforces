  <name>Task 1: 002_elo_function.sql — calculate_elo() SECURITY DEFINER</name>
  <files>supabase/migrations/002_elo_function.sql</files>
  <action>
    Create `supabase/migrations/002_elo_function.sql`. The file must be idempotent (CREATE OR REPLACE).

    Add a private helper `get_rank_tier(elo INT)` that returns TEXT using Codeforces-style tiers matching EloRank.kt: 0–1199 → 'Newbie', 1200–1399 → 'Pupil', 1400–1599 → 'Specialist', 1600–1899 → 'Expert', 1900–2099 → 'Candidate Master', 2100–2299 → 'Master', 2300–2399 → 'International Master', 2400–2599 → 'Grandmaster', 2600–2899 → 'International Grandmaster', 2900+ → 'Legendary Grandmaster'.

    Create `public.calculate_elo(p_user_id UUID, p_correct INT, p_total INT, p_time_ms BIGINT) RETURNS JSON` as `LANGUAGE plpgsql SECURITY DEFINER SET search_path = public`.

    Function body logic:
    - Declare variables: `v_current_elo INT`, `v_k_factor INT`, `v_performance FLOAT`, `v_time_factor FLOAT`, `v_delta INT`, `v_new_elo INT`, `v_rank_tier TEXT`.
    - SELECT `elo_score` INTO `v_current_elo` FROM `user_progress` WHERE `user_id = p_user_id`; if NOT FOUND set `v_current_elo = 1000`.
    - `v_performance = p_correct::FLOAT / NULLIF(p_total, 0)` (default 0 if null).
    - `v_time_factor = CASE WHEN p_total > 0 AND (p_time_ms / p_total) < 5000 THEN 0.1 ELSE 0.0 END`.
    - `v_k_factor = CASE WHEN v_current_elo < 1200 THEN 40 WHEN v_current_elo < 2100 THEN 32 ELSE 24 END`.
    - `v_delta = ROUND(v_k_factor * (COALESCE(v_performance, 0) + v_time_factor - 0.5))::INT`.
    - `v_new_elo = GREATEST(0, v_current_elo + v_delta)`.
    - `v_rank_tier = get_rank_tier(v_new_elo)`.
    - UPDATE `user_progress` SET `elo_score = v_new_elo`, `updated_at = NOW()` WHERE `user_id = p_user_id`.
    - Upsert `leaderboard`: INSERT INTO leaderboard (user_id, username, elo_score, weekly_elo, rank_tier, updated_at) SELECT p_user_id, u.username, v_new_elo, GREATEST(0, l.weekly_elo + v_delta), v_rank_tier, NOW() FROM users u LEFT JOIN leaderboard l ON l.user_id = p_user_id WHERE u.id = p_user_id ON CONFLICT (user_id) DO UPDATE SET elo_score = EXCLUDED.elo_score, weekly_elo = GREATEST(0, leaderboard.weekly_elo + v_delta), rank_tier = EXCLUDED.rank_tier, updated_at = EXCLUDED.updated_at.
    - RETURN `json_build_object('new_elo', v_new_elo, 'rank_tier', v_rank_tier, 'elo_delta', v_delta)`.

    Grant EXECUTE on both functions to `authenticated` role:
    `GRANT EXECUTE ON FUNCTION public.get_rank_tier(INT) TO authenticated;`
    `GRANT EXECUTE ON FUNCTION public.calculate_elo(UUID, INT, INT, BIGINT) TO authenticated;`

    Add a migration header comment block: date 2026-07-23, description "ELO calculation SECURITY DEFINER function".
  </action>

  <files>supabase/migrations/003_streak_function.sql</files>
  <action>
    Create `supabase/migrations/003_streak_function.sql`. Idempotent throughout.

    Section 1 — streak_history table:
    CREATE TABLE IF NOT EXISTS `public.streak_history` with columns: `user_id UUID NOT NULL REFERENCES public.users(id) ON DELETE CASCADE`, `practiced_date DATE NOT NULL`, PRIMARY KEY (user_id, practiced_date).
    Enable RLS: `ALTER TABLE public.streak_history ENABLE ROW LEVEL SECURITY`.
    Add read policy `streak_history_select_own` FOR SELECT USING `user_id = auth.uid()`.
    No direct INSERT policy — inserts happen only via the SECURITY DEFINER function below.

    Section 2 — update_streak() function:
    Create `public.update_streak(p_user_id UUID, p_today_date DATE) RETURNS JSON` as `LANGUAGE plpgsql SECURITY DEFINER SET search_path = public`.

    Declare variables: `v_last_date DATE`, `v_streak INT`, `v_freeze INT`, `v_days_diff INT`, `v_was_freeze BOOLEAN := FALSE`.

    SELECT `last_practice_date`, `streak_count`, `streak_freeze_count` INTO `v_last_date`, `v_streak`, `v_freeze` FROM `user_progress` WHERE `user_id = p_user_id`; if NOT FOUND set `v_last_date = NULL, v_streak = 0, v_freeze = 0`.

    Compute `v_days_diff = p_today_date - v_last_date` (NULL-safe: treat NULL as 999).

    Logic (using IF/ELSIF/ELSE):
    - IF v_days_diff = 0: no-op (already practiced today).
    - ELSIF v_days_diff = 1: v_streak = v_streak + 1.
    - ELSIF v_days_diff > 1 AND v_freeze > 0: v_freeze = v_freeze - 1; v_was_freeze = TRUE; (streak unchanged).
    - ELSE: v_streak = 1.

    Always (when v_days_diff != 0): UPDATE `user_progress` SET `streak_count = v_streak`, `streak_freeze_count = v_freeze`, `last_practice_date = p_today_date`, `updated_at = NOW()` WHERE `user_id = p_user_id`.

    Always: INSERT INTO `streak_history` (user_id, practiced_date) VALUES (p_user_id, p_today_date) ON CONFLICT DO NOTHING.

    RETURN `json_build_object('streak_count', v_streak, 'streak_freeze_available', v_freeze > 0, 'was_freeze_used', v_was_freeze)`.

    Grant EXECUTE to `authenticated`:
    `GRANT EXECUTE ON FUNCTION public.update_streak(UUID, DATE) TO authenticated;`

    Add migration header comment: date 2026-07-23, description "Streak tracking function + history table".
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → calculate_elo() | Client sends game metrics (correct, total, time_ms) — never ELO delta |
| client → update_streak() | Client sends today's UTC date — server validates gap logic |
| Edge Function → service_role | refresh-streak-freeze uses service_role key stored in Supabase env vars |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | calculate_elo params | mitigate | SECURITY DEFINER + RLS: client cannot write elo_score directly; function reads current ELO from DB, not from client |
| T-03-02 | Tampering | update_streak p_today_date | mitigate | Server uses p_today_date only for gap computation; streak_count cannot exceed days_diff=1 per call; no multi-day jumps via single call |
| T-03-03 | Elevation of Privilege | streak_history INSERT | mitigate | No direct INSERT RLS policy; only SECURITY DEFINER function may write — client cannot forge historical dates |
| T-03-04 | Spoofing | refresh-streak-freeze auth | mitigate | Function uses SUPABASE_SERVICE_ROLE_KEY from Supabase env vars — never exposed to client |
| T-03-SC | Tampering | npm/pip/cargo installs | accept | No package manager installs in this plan — pure SQL + Deno Edge Function with CDN imports |
</threat_model>
