<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User input → AuthViewModel | Email/password entered by user — untrusted; crosses into ViewModel |
| AuthViewModel → Supabase HTTPS | Credentials transmitted to Supabase over TLS |
| Google OAuth → Deep Link | OAuth authorization code arrives via vietforces://login scheme |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Spoofing | AuthRepositoryImpl.signIn | mitigate | Server-side validation only; client sends raw credentials over TLS, supabase-kt handles token validation |
| T-02-02 | Information Disclosure | Password fields in LoginScreen / RegisterScreen | mitigate | Both fields use PasswordVisualTransformation(); no password is logged, stored locally, or placed in error messages |
| T-02-03 | Tampering | vietforces://login deep link intent | mitigate | supabase.handleDeeplinks verifies the OAuth state parameter; MainActivity uses launchMode=singleTask to prevent intent spoofing via task stack |
| T-02-04 | Elevation of Privilege | signOut in AuthViewModel | mitigate | Calls supabase.auth.signOut() which revokes the server-side refresh token, not just clears local storage |
| T-02-05 | Information Disclosure | AuthRepositoryImpl token storage | accept | supabase-kt stores session tokens in Android Keystore-backed EncryptedSharedPreferences internally; no manual token handling |
| T-02-SC | Tampering | No new package installs in this plan | accept | lifecycle-runtime-compose is already in the androidx ecosystem; no third-party install; slopcheck not needed |
</threat_model>
